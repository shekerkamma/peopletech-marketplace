# 00-account-briefing

Orchestrator skill — chains four phases to produce a stakeholder-ready pre-meeting briefing for a named enterprise account. Built for Sheker's automotive POC factory; default account is TMNA but works for any account folder under `accounts/`.

## When this fires

- Sheker says any of: *"brief me on TMNA"*, *"prep for the TMNA call"*, *"give me the latest on TMNA"*, *"/account-briefing TMNA"*.
- Substitute any account name that has a folder at `accounts/<ACCOUNT>/`.

## Inputs
- `ACCOUNT` — the account folder name (e.g. `TMNA`). If not provided, ask for it once.
- Optional: `MEETING_TOPIC` — if Sheker specifies what the meeting is about, weight the briefing toward that.

## Output
A markdown file at:
```
accounts/<ACCOUNT>/projects/briefings/<YYYY-MM-DD>-pre-meeting.md
```
Plus today's daily note gets a one-line log of the briefing generation.

## Phases (Contract Pattern)

Each phase produces a structured JSON return + a human-readable section that becomes part of the final briefing.

### Phase 1 — Discovery
**Action:** Read `accounts/<ACCOUNT>/CLAUDE.md`. List files in `accounts/<ACCOUNT>/projects/` (last 14 days, sorted newest first). Read `MEMORY.md` and extract the `## Open threads` section. Extract any `## Active workstreams` items that mention the account.

**Returns:**
```json
{
  "account_context": "summary of accounts/<ACCOUNT>/CLAUDE.md",
  "active_pocs": ["file paths from accounts/<ACCOUNT>/projects/, newest first"],
  "open_threads": ["bullets from MEMORY.md → Open threads"],
  "active_workstreams_mentioning_account": ["..."]
}
```

If `accounts/<ACCOUNT>/CLAUDE.md` doesn't exist: stop, suggest:
```ps1
mkdir accounts/<ACCOUNT>; mkdir accounts/<ACCOUNT>/projects/briefings; New-Item accounts/<ACCOUNT>/CLAUDE.md
```

### Phase 2 — Notion pull
**Action:** Hydrate `NOTION_BEARER_AUTH` from User env. Read the `## Search aliases` section from `accounts/<ACCOUNT>/CLAUDE.md` if present — search Notion for **each alias** and merge+dedupe results (by page ID). If no aliases section exists, default to a single search on `<ACCOUNT>`.

```ps1
$env:NOTION_BEARER_AUTH = [Environment]::GetEnvironmentVariable('NOTION_BEARER_AUTH','User')
# For each alias (e.g. "TMNA", "Toyota", "Toyota Motor North America"):
& "$env:USERPROFILE\go\bin\notion-pp-cli.exe" search "<alias>" --agent --limit 30
```

Filter results to: pages edited in the last 14 days; pages whose title or excerpt mentions any alias. Pull both meeting notes and project documentation. If the parent databases ("Meeting Notes", "Project Documentation") are accessible, prefer fresh entries from those.

**Returns:**
```json
{
  "recent_meetings": [{"title": "...", "url": "...", "last_edited": "...", "excerpt": "..."}],
  "recent_docs":     [{"title": "...", "url": "...", "last_edited": "...", "excerpt": "..."}]
}
```

If Notion search returns 0 hits for the account name: continue with empty arrays, note in the briefing that Notion has no recent activity tagged with this account.

### Phase 3 — Synthesize
**Action:** Read `voice-profile.md` and apply its rules to all prose generated. Read `accounts/<ACCOUNT>/CLAUDE.md` "Tone for <ACCOUNT> deliverables" section if present.

Build the briefing with this structure:

```markdown
# <ACCOUNT> pre-meeting briefing — <DATE>

> One-page brief. Generated by /account-briefing.

## Where we are
2-3 sentences. What's the current state of the engagement. Pull from account_context + active_pocs.

## Since last touch
3-5 bullets. Most-recent Notion meetings + docs (Phase 2 output), each one line.

## Open threads
Bullets from open_threads — anything Sheker said he was waiting on.

## What to drive in this meeting
**4 prioritized talking points** (never 3 — `voice-profile.md` bans the rule of three). Tie each one back to a business outcome the account cares about.

## Risks / watchouts
Anything from the account's CLAUDE.md compliance defaults that could bite (e.g. for TMNA: US data residency, no cross-cloud without approval).

## Quick references
- Account CLAUDE.md: accounts/<ACCOUNT>/CLAUDE.md
- Recent Notion edits: [list with URLs]
```

**Returns:**
```json
{
  "briefing_markdown": "<full markdown body>",
  "talking_points":    ["bullet 1", "bullet 2", "bullet 3"]
}
```

### Phase 4 — Deliver
**Action:**
1. Ensure `accounts/<ACCOUNT>/projects/briefings/` exists (`mkdir -p` if missing — first run for a new account will need this).
2. Write the briefing to `accounts/<ACCOUNT>/projects/briefings/<YYYY-MM-DD>-pre-meeting.md` (overwrite if exists).
3. Append one line to today's `memory/<YYYY-MM-DD>.md`:
   ```
   - Account briefing generated: <ACCOUNT> → accounts/<ACCOUNT>/projects/briefings/<DATE>-pre-meeting.md (talking points: ...)
   ```
4. Print the file path and the 4 talking points to the user.

**Returns:**
```json
{
  "file_path": "accounts/<ACCOUNT>/projects/briefings/<YYYY-MM-DD>-pre-meeting.md",
  "talking_points": ["...", "...", "..."]
}
```

## How to run this skill efficiently

For a first-pass implementation, run all four phases in the same conversation context. The Contract Pattern is documented so the chain is *legible* and ready to be split into isolated sub-agents (via `Agent` tool with `isolation: "worktree"` per phase) when Sheker decides he wants that level of context isolation.

## Failure modes
- Account folder missing → suggest the mkdir command, stop.
- Notion CLI not authed → continue without Notion section, note it in the briefing.
- Notion returns 0 hits → continue with empty section, note "no recent Notion activity for <ACCOUNT>".
- All four phases succeed but briefing is empty → stop, tell Sheker that we found nothing fresh to brief on — don't manufacture filler.

## Honest constraints
- This skill reads `voice-profile.md` and tries to apply it. Voice quality depends on how well that file captures Sheker's tone. If briefings come back sounding generic, fix `voice-profile.md` first, not this skill.
- The "talking points" are inferred from recent activity + account context. They are starting points, not strategy. Sheker should redirect when the auto-derived points miss the actual meeting purpose.
